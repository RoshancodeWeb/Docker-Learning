1 The Big Picture

Steps 1 and 2 ran one container at a time, by hand. Step 3 is the jump to many services running together and talking to each other — started with a single command. That orchestration is what Docker Compose does.

Step 1:  docker run nginx          → ONE ready-made container
Step 2:  docker build → docker run → ONE container (my own image)
Step 3:  docker compose up         → MANY containers, wired together, ONE command
🎼 The one-sentence definition Docker Compose is a conductor: I describe all my services in one file (docker-compose.yml), then docker compose up builds them, puts them on a shared private network, and starts them all in the right order.

2 What We're Building: A Visit Counter

The demo is a tiny web page that counts visitors. The clever part: the count is not stored in my app — it's stored in Redis, a separate container. So even if my app restarts, the number is remembered, because Redis is holding it.

Two services work together:

index.js — the two lines that matter most

// Connect to Redis by the HOSTNAME "redis" — that's the service
// name from docker-compose.yml, NOT an IP. Docker's network
// turns that name into the right address automatically.
const redisClient = createClient({
  url: process.env.REDIS_URL || "redis://redis:6379",
});

// Connect to Redis FIRST, then start accepting web requests.
async function start() {
  await redisClient.connect();          // ⬅ waits for Redis
  app.listen(PORT, "0.0.0.0", () => { ... });  // ⬅ only then listens
}
💡 Why "0.0.0.0" and not localhost? 0.0.0.0 means "accept connections from outside the container," not just from inside it. If the app bound to localhost only, Docker's port forwarding couldn't reach it. It's the standard for containerized apps.

4 The Dockerfile, Line by Line

Docker reads this top-to-bottom; each instruction adds a layer to the image.

FROM node:20-alpine        # start from a tiny Linux that already has Node v20
WORKDIR /app               # work inside /app (a clean room for my files)
COPY package*.json ./      # copy ONLY the dependency list first (for caching)
RUN npm install            # install deps — happens once, while BUILDING
COPY . .                   # now copy the rest of my code in
EXPOSE 3000                # documentation only — does NOT open the port
CMD ["node","index.js"]   # the command that runs when a CONTAINER starts
DistinctionWhat it means
RUN vs CMDRUN happens once, while building the image (e.g. install). CMD happens later, every time a container starts (e.g. launch the app). "Set up the kitchen" vs "cook when a customer arrives."
EXPOSE vs ports:EXPOSE is just a label for humans. The real door-opening happens in the compose file's ports:. EXPOSE = the sign on the door; ports: = actually unlocking it.
Why COPY package*.json first?Code changes constantly; dependencies rarely. Copying the dependency list and installing before copying code means editing index.js doesn't re-run the slow npm install — Docker reuses the cached layer.
🚫 The gotcha that cost me: a missing CMD My first Dockerfile ended at EXPOSE 3000 with no CMD. Result: web exited with code 0 and the page refused to load. With no CMD, Docker ran the base image's default (the Node REPL), which had nothing to do, so it quit instantly — no main process means the container stops. The fix: add CMD ["node","index.js"] so the container has a long-lived process (the web server) and stays up.

5 docker-compose.yml, Line by Line

⚠️ It's YAML — indentation IS the structure Things indented further "belong to" the thing above them. There are no brackets; the spacing is the meaning. Read it like an outline, and keep the indentation consistent. 
services:                      # the list of containers I want
  web:                         # service #1 — MY app (the name is my choice)
    build: .                    # build the image from the Dockerfile in this folder
    ports:
      - "3000:3000"            # HOST:CONTAINER — a door from my Mac into the app
    environment:
      REDIS_URL: redis://redis:6379   # config; "redis" = the service name below
    depends_on:
      - redis                  # start redis BEFORE web

  redis:                       # service #2 — the data store (ready-made)
    image: redis:7              # pull official Redis v7; NO build
    volumes:
      - redis-data:/data       # attach the named volume to Redis's data folder
                               # (no ports: → reachable only INSIDE the network)

volumes:                       # top-level: declare the named volume
  redis-data:
KeyMeaning
services:The list of containers to run
web: / redis:Service names I choose — each one becomes a hostname on the network
build: .Build this service's image from the Dockerfile here
image: redis:7Pull a ready-made image instead of building
ports:Publish a door to the outside (host:container) — only for services the browser needs
environment:Pass config (env vars) into the container
depends_on:Startup order — start the listed service first
volumes: (top-level)Declare a named volume so data can outlive the container
⚠️ depends_on caveat It waits for Redis to start, not to be fully ready for connections. Usually fine for Redis. Fussy services need a healthcheck to wait until they're truly ready.

6 Networking — Find Each Other by Name

🌟 The heart of Step 3 Compose creates a private network where every service is reachable by its service name. My app connects to redis://redis:6379 — that redis is the service name, not an IP address. No IP addresses, ever. Docker resolves the name automatically.

This works in all directions: redis could reach the app at web:3000; a future frontend could reach a backend at http://backend:8000.

Internal port vs published port (a key subtlety)

From…Address usedWhich portNeeds ports:?
Another containerredis:6379 / web:3000the container port✗ No
My Mac's browserlocalhost:3000the host port✓ Yes

Proof: redis has no ports: line, yet web still reaches it — because container-to-container traffic doesn't need publishing. ports: is only for letting the outside world in. So: only publish ports for services the browser actually uses; keep your database private. 🔒

7 Ports — host : container

A port mapping reads left = my Mac, right = inside the container:

- "3000:3000" │ │ │ └── port INSIDE the container (where my app listens) └──────── port on MY MAC (what I type in the browser)

The two numbers don't have to match. "8080:3000" would mean "visit localhost:8080, and it reaches the app's 3000 inside." Whatever's on the left is the one you open in the browser.

🚫 The typo that gave me "connection refused" I wrote "30000:3000" by accident (an extra 0). The containers were perfectly healthy, but my app was published on my Mac's port 30000, not 3000 — so localhost:3000 refused. How I found it: docker compose ps showed 0.0.0.0:30000->3000 in the PORTS column. Lesson: the PORTS column tells you the real doorway — never trust your memory of what you typed.

8 Data Persistence — Named Volumes

By default Redis saves its data to /data inside the container — so a docker compose down destroys the container and the count. A named volume moves that folder to a managed box outside the container, so it survives.

WITHOUT a volume: WITH redis-data:/data: ┌─ redis container ─┐ ┌─ redis container ─┐ │ /data │ │ /data ──────────┼──┐ plugs in │ (dies on `down`) │ └───────────────────┘ │ └───────────────────┘ ▼ ┌─ redis-data (managed box) ─┐ │ dump.rdb (survives!) ✓ │ └────────────────────────────┘

The two volumes: keys do different jobs:

WhereWhat it does
volumes: inside redis:Attaches — "mount the box redis-data at /data in this container." Reads as NAME:PATH.
volumes: at the top levelDeclares — "create a managed storage box called redis-data." Like declaring a variable.
✅ The survival ladder
Where the data livesapp restartstop / startdown
App's memory✗✗✗
Redis /data (in container)✓✓✗
Named volume✓✓✓
💡 Named volume vs bind mount redis-data:/data (a name, no slash) is Docker-managed storage — best for databases. ./src:/app/src (a path, with ./) maps a real folder from my Mac into the container — great for live code reload in development.

9 The Commands I Actually Use

Run all of these from the folder that has the docker-compose.yml.

docker compose up              # build + network + start, logs in the foreground
docker compose up -d           # -d = detached (background), frees my terminal
docker compose up --build      # force a rebuild first (after a Dockerfile/code change)
docker compose down            # stop AND remove containers + network (keeps named volumes)
docker compose down -v         # also wipe volumes — DELETES the data (careful!)
docker compose ps              # list this project's services + their PORTS
docker compose restart web     # bounce ONE service (does NOT re-read the yaml)
docker compose logs -f web      # follow one service's logs live
docker compose exec web sh      # open a shell INSIDE the running web container

restart vs up vs up --build — when to use which

I changed…CommandWhy
Nothing — just want a clean bouncedocker compose restart webSame container, off and on again
The docker-compose.yml (ports, volumes, env)docker compose up -dup compares the file to reality and recreates what changed
My code or the Dockerfiledocker compose up -d --buildRebuilds the image first, then recreates
🚫 restart does NOT apply compose-file edits I changed the yaml and ran restart — nothing happened. restart just stops and starts the same existing container with its old config; it never re-reads the file. To apply edits (like adding the volume) you need up, which recreates the container. Mental model: restart = off/on; up = make reality match the file.

Viewing logs — three ways

GoalCommand
One Compose service, livedocker compose logs -f web
Any container anywhere (incl. docker run)docker logs -f <container-name>
Log files an app writes inside itselfdocker compose exec web sh → then cat / tail
💡 Naming containers, and where logs come from Add --name myname to a plain docker run to label a container (otherwise Docker invents one like magical_franklin). With Compose, names are auto-built as project-service-1, e.g. roshan-prac-step3-web-1. My app logs with console.log, which goes to stdout — Docker captures that automatically, so docker compose logs shows it; there's no log file to hunt for inside.

10 Shutdown — Signals & Exit Codes

Ctrl+C (or docker compose down) triggers a graceful, two-phase shutdown:

  • SIGTERM (signal 15) = "please shut down nicely" — Docker waits ~10 seconds.
  • SIGKILL (signal 9) = "die now" — only if it didn't stop in time (forced).
ContainerHandled SIGTERM?How it diedExit code
redis✓ yes — saved data, exited itselfclean0
web (my app)✗ no — ignored itforce-killed after timeout137
💡 Exit codes encode the signal 137 = 128 + 9 (killed by SIGKILL). 143 = 128 + 15 (SIGTERM). Exit 0 = finished cleanly. Optional polish — make my app shut down gracefully like Redis: 
process.on("SIGTERM", () => server.close(() => process.exit(0)));

11 Gotchas I Actually Hit (Field Notes)

🚫 #1 — "connection refused" from a plain docker run I ran the app alone with docker run and got getaddrinfo ENOTFOUND redis. Cause: the hostname redis only exists inside a Compose network. My code waits for redisClient.connect() before app.listen() — so with no Redis, it never starts the web server, and nothing listens on 3000. Fix: run it with docker compose up (which starts Redis too), not a standalone docker run.
🚫 #2 — wrong host port (30000:3000) A typo published the app on port 30000, not 3000. The container was healthy; only the doorway was wrong. Found it in the PORTS column of docker compose ps. Confirmed with curl localhost:3000 (curl never caches, unlike a browser).
🚫 #3 — edited the yaml, ran restart, nothing changed restart reuses the old container config. Compose-file edits need docker compose up to take effect.
✅ My debugging ladder for "connection refused" 1. docker compose ps — are the containers even running?
2. Look at the PORTS column — what port is actually published?
3. curl http://localhost:<port> — get the honest answer (no browser caching).
4. If a container exited — docker compose logs <service> to see why.

12 Full Command Cheat Sheet

# --- lifecycle ---
docker compose up                 # build + start, logs in foreground
docker compose up -d              # start in background
docker compose up -d --build      # rebuild image, then start
docker compose down               # stop + remove (keeps volumes)
docker compose down -v            # stop + remove + WIPE volumes
docker compose restart web        # bounce one service
docker compose stop / start       # stop/start without removing

# --- inspecting ---
docker compose ps                 # services + status + ports
docker compose logs -f web        # follow one service's logs
docker compose logs --tail 50     # last 50 lines of everything
docker compose exec web sh        # shell into the running container
docker compose exec redis redis-cli   # open the Redis CLI to inspect the count

# --- the data, hands-on ---
docker compose exec redis redis-cli GET visits   # read the stored count
docker volume ls                  # list named volumes on my machine
